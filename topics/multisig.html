1. Overview of Multisig Approaches

Bitcoin multisig has evolved through four distinct eras, each improving on privacy, efficiency, and flexibility. The core idea is the same: require M-of-N keys to authorize a spend. How that requirement is expressed on-chain has changed significantly.

Bare Multisig ──▶ P2SH Multisig ──▶ P2WSH Multisig ──▶ Taproot Multisig (2009) (2012) (2017) (2021) BIP 11 BIP 16 BIP 141 BIP 340/341/342 Deprecated Widely used Current standard Emerging

2. Bare Multisig (OP_CHECKMULTISIG)

The Original Approach Deprecated

The very first multisig implementation. The entire script — including all public keys — is placed directly in the transaction output's scriptPubKey.

How It Worked

The locking script contained all keys and the threshold in plain sight:

scriptPubKey:  OP_2 <PubKeyA> <PubKeyB> <PubKeyC> OP_3 OP_CHECKMULTISIG
scriptSig:     OP_0 <SigA> <SigB>

The OP_0 at the start of the scriptSig is required due to a long-standing off-by-one bug in OP_CHECKMULTISIG that pops one extra element from the stack.

Why It's No Longer Used

  • UTXO bloat: All public keys sit in the UTXO set, which every full node must hold in memory.
  • Non-standard: Since Bitcoin Core ~v0.10 (2014–15), nodes won't relay bare multisig transactions by default.
  • No privacy: The spending policy is fully visible before any spend occurs.

BIP 11 Defined standardness rules for bare multisig.

3. P2SH Multisig

Pay-to-Script-Hash Widely Used

Activated in 2012, P2SH wraps the multisig script behind a hash. The output only contains a 20-byte hash — no public keys visible until spend time. Addresses start with 3.

How It Works

scriptPubKey:  OP_HASH160 <Hash160(redeemScript)> OP_EQUAL

redeemScript:  OP_2 <PubKeyA> <PubKeyB> <PubKeyC> OP_3 OP_CHECKMULTISIG

scriptSig:     OP_0 <SigA> <SigB> <redeemScript>

Nodes hash the provided redeem script, check it matches the output, then execute the script. The spending policy is only revealed when the UTXO is spent.

Pros & Cons

  • Pro: Moves script data out of the UTXO set.
  • Pro: Policy hidden until spend time.
  • Pro: Universal wallet and exchange support.
  • Con: Redeem script limited to 520 bytes (~15-of-15 max).
  • Con: Full script revealed at spend time.
  • Con: No witness discount — full-weight signature data.

BIP 16 P2SH   BIP 13 P2SH address format

4. P2WSH Multisig (SegWit)

Pay-to-Witness-Script-Hash Current Standard

The SegWit v0 version, activated August 2017. Uses a 32-byte SHA256 hash and moves all signature data into the witness. Addresses use bech32 encoding, starting with bc1q.

How It Works

scriptPubKey:  OP_0 <SHA256(witnessScript)>

witness:       OP_0  <SigA>  <SigB>  <witnessScript>

witnessScript: OP_2 <PubKeyA> <PubKeyB> <PubKeyC> OP_3 OP_CHECKMULTISIG

Improvements Over P2SH

  • Witness discount: Witness data is counted at ¼ weight — roughly 30–50% fee savings for multisig.
  • Larger scripts: Witness scripts can be up to 10,000 bytes (standard policy), removing the 520-byte limit.
  • Malleability fix: Witness data is excluded from the txid, solving transaction malleability.
  • Stronger hash: SHA256 (32 bytes, 128-bit collision resistance) vs HASH160 (20 bytes, 80-bit).

BIP 141 SegWit   BIP 143 Signature hashing   BIP 173 Bech32 addresses

5. Taproot Key-Path Multisig (MuSig2 / FROST)

Schnorr Aggregate Signatures Emerging

With Taproot (SegWit v1, November 2021), multiple signers can collaboratively produce a single public key and a single signature. On-chain, the spend is indistinguishable from a regular single-key payment. Addresses start with bc1p.

MuSig2 (N-of-N)

All N signers combine their keys into one aggregate key. Spending requires a two-round signing ceremony:

Round 1: Each signer generates + shares nonce commitments Round 2: Each signer produces a partial signature ──▶ Partial sigs combine into ONE valid Schnorr signature

The result: a single 32-byte key in the output, a single 64-byte signature to spend. No one can tell it was multisig.

FROST (T-of-N Threshold)

Flexible Round-Optimized Schnorr Threshold signatures. Any T of N signers can produce a single aggregate signature — without all N participating. Uses distributed key generation (DKG) and Shamir-style secret sharing at the nonce/signing level.

On-Chain Footprint

scriptPubKey:  OP_1 <32-byte x-only pubkey>
witness:       <64-byte Schnorr signature>

Identical to any single-sig Taproot spend. Maximum privacy.

Trade-offs

  • Interactive: Signers must be online simultaneously for the signing ceremony.
  • MuSig2 has a finalized BIP and early production use (e.g., BitGo 2-of-3).
  • FROST has working implementations but no finalized Bitcoin BIP yet.

BIP 340 Schnorr   BIP 341 Taproot   BIP 327 MuSig2

6. Taproot Script-Path Multisig (OP_CHECKSIGADD)

MAST + Tapscript Available

When key-path spending isn't possible (not all signers available, no FROST setup), Taproot allows fallback to script-based multisig using a Merkleized script tree.

How the Script Tree Works

Taproot Output ┌───────┴───────┐ Internal Key Merkle Root ┌─────┴─────┐ Leaf A Leaf B (A + B) (A + C) ┌───┴───┐ Leaf C Leaf D (B + C) (timelock)

Each leaf can hold a different spending condition. To spend via script-path, you reveal only the leaf you're using plus a Merkle proof. Other leaves stay hidden.

OP_CHECKSIGADD

Replaces the legacy OP_CHECKMULTISIG (which is disabled in tapscript). Checks one key at a time and accumulates a counter:

<PubKeyA> OP_CHECKSIG
<PubKeyB> OP_CHECKSIGADD
<PubKeyC> OP_CHECKSIGADD
OP_2 OP_EQUAL

Each signer provides a valid signature or empty value. No dummy OP_0 bug. Each signature maps to exactly one key.

Key-Path vs Script-Path

  • Key-path: Preferred. Cheaper, more private, smaller footprint.
  • Script-path: Fallback for when interactive signing isn't feasible, or for complex policies (timelocks, recovery paths).
  • Common pattern: Key-path for the happy case (e.g., 2-of-2 MuSig2), script-path leaves for fallback scenarios.

BIP 341 Taproot / MAST   BIP 342 Tapscript   BIP 340 Schnorr

6b. Five Taproot Multisig Patterns

Jimmy Song's Taproot Multisig presentation (Advancing Bitcoin 2022) lays out five distinct patterns, progressing from simplest to most sophisticated. Each trades off complexity for privacy and fee savings.

Pattern 1: Single Leaf k-of-n TapScript

The simplest Taproot multisig — closest to traditional P2WSH. One tapscript leaf holds a single k-of-n script using OP_CHECKSIGADD.

Taproot Output ├── Internal Key (unspendable or MuSig aggregate) └── Script Tree └── Leaf: <PubA> CHECKSIG <PubB> CHECKSIGADD <PubC> CHECKSIGADD 2 EQUAL
  • Pro: Non-interactive — signers can be chosen on the fly, just like legacy multisig.
  • Con: Reveals all pubkeys on-chain at spend time. Larger on-chain footprint.
  • Reality: What most hardware wallets implement first, because it's closest to what they already know.

Pattern 2: Multi-Leaf k-of-k TapScript

Instead of one leaf with all keys, create a separate leaf for each valid signer combination. For a 2-of-3, that's three leaves: (A+B), (A+C), (B+C).

Taproot Output ├── Internal Key └── Script Tree ├── Leaf 1: <PubA> CHECKSIGVERIFY <PubB> CHECKSIG ├── Leaf 2: <PubA> CHECKSIGVERIFY <PubC> CHECKSIG └── Leaf 3: <PubB> CHECKSIGVERIFY <PubC> CHECKSIG
  • Pro: Only the spending leaf's pubkeys are revealed — unused combinations stay hidden.
  • Pro: Smaller per-spend footprint than single-leaf (fewer keys revealed).
  • Con: Signers must be predetermined. Number of leaves grows combinatorially: C(n,k).

Pattern 3: Multi-Leaf MuSig TapScript

Same multi-leaf structure, but each leaf uses a MuSig aggregate key instead of listing individual keys. The leaf contains a single pubkey and requires a single signature.

Taproot Output ├── Internal Key └── Script Tree ├── Leaf 1: <MuSig(A,B)> CHECKSIG ├── Leaf 2: <MuSig(A,C)> CHECKSIG └── Leaf 3: <MuSig(B,C)> CHECKSIG
  • Pro: Maximum privacy — individual pubkeys are never revealed on-chain.
  • Pro: Smallest on-chain footprint: one key, one signature per spend.
  • Con: Requires interactive nonce exchange (MuSig2 signing ceremony).

Pattern 4: "Everything" Multisig

Combine all the above in a single Taproot address. Different leaves use different strategies, allowing dynamic selection based on signer availability and privacy preference.

Taproot Output ├── Key Path: MuSig(A,B,C) ← n-of-n, best case └── Script Tree ├── MuSig leaves (A+B), (A+C), (B+C) ← k-of-k MuSig ├── k-of-k bare leaves ← fallback if MuSig unavailable └── Single leaf k-of-n ← last resort
  • Key-path (MuSig n-of-n) is cheapest and most private — used when all signers cooperate.
  • MuSig leaves used when only k signers are available and they can interact.
  • Bare leaves as fallback when interaction isn't possible.
  • MAST efficiency: additional tree levels cost minimal extra data.

Pattern 5: Degrading Multisig with Timelocks

Add timelocked fallback branches that require fewer signatures as time passes. Enables inheritance planning and key-loss recovery.

Taproot Output ├── Key Path: MuSig(A,B,C) ← normal spend (3-of-3) └── Script Tree ├── 2-of-3 leaves ← available immediately ├── 1-of-3 leaves + OP_CSV 52560 ← after ~1 year └── Recovery key + OP_CSV 105120 ← after ~2 years
  • Use case: If you lose a key, wait for the timelock and spend with fewer keys.
  • Inheritance: Heirs can access funds after a defined period without needing all keys.
  • Important: Requires periodic fund movement to reset timelocks and prevent security degradation.
  • Uses OP_CHECKSEQUENCEVERIFY (BIP 112) for relative timelocks.

7. P2SH Multisig Deep Dive

P2SH multisig remains the most widely deployed and understood form of Bitcoin multisig. Here's how it works end to end.

Setup

  1. Generate keys: Each participant generates a keypair (typically on a hardware wallet). Only the public key is shared.
  2. Agree on parameters: Decide on M (threshold) and N (total keys), and the key ordering. BIP 67 defines lexicographic key sorting for deterministic multisig.
  3. Build the redeem script: 
    OP_2 <PubKeyA> <PubKeyB> <PubKeyC> OP_3 OP_CHECKMULTISIG
  4. Hash it: HASH160(redeemScript) → 20-byte hash.
  5. Derive the address: Base58Check encode with version byte 0x05 → starts with 3.
  6. Back up: Each participant stores the redeem script (or all public keys + parameters to reconstruct it). Losing the redeem script means losing access to the funds.

Receiving

Receiving is straightforward — share the P2SH address. The sender sees a normal-looking address starting with 3. They have no way of knowing it's multisig, what threshold is required, or how many keys are involved.

Spending

  1. Create the transaction: Build an unsigned transaction spending from the P2SH UTXO. This is typically done in the coordinator wallet (e.g., Sparrow).
  2. Export as PSBT: The unsigned transaction is serialized as a Partially Signed Bitcoin Transaction (BIP 174 PSBT).
  3. Collect signatures: Pass the PSBT to M of the N signers. Each signs with their key and returns the PSBT with their partial signature added.
  4. Combine and finalize: The coordinator combines all partial PSBTs. Once M signatures are present, the transaction is finalized — the scriptSig is assembled: 
    OP_0 <SigA> <SigB> <redeemScript>
  5. Broadcast: The finalized transaction is broadcast to the network.

Key Rotation

P2SH multisig has no native key rotation mechanism. To rotate a key:

  1. Create a new multisig setup with the updated key set.
  2. Send all funds from the old multisig address to the new one.
  3. This is a normal on-chain transaction — it costs fees and is publicly visible.

This is one area where Taproot improves things: a key-path spend with MuSig2 makes the rotation transaction look like any other payment.

8. Output Descriptors

What Is an Output Descriptor?

An output descriptor is a human-readable string that fully describes how to derive a set of Bitcoin addresses and how to spend from them. Introduced in Bitcoin Core 0.17 (2018), they replace the older "import a key/address" model with a structured, portable format.

A descriptor encodes:

  • The script type (P2SH, P2WSH, P2TR, etc.)
  • The keys involved (xpubs, xprvs, or raw keys)
  • The derivation paths
  • The threshold (for multisig)
  • A checksum for integrity

Descriptor Syntax

# Single-sig P2WPKH
wpkh([fingerprint/derivation]xpub.../0/*)

# 2-of-3 P2SH multisig
sh(multi(2,
  [fingerprint1/path1]xpub1.../0/*,
  [fingerprint2/path2]xpub2.../0/*,
  [fingerprint3/path3]xpub3.../0/*
))

# 2-of-3 P2WSH multisig (SegWit)
wsh(sortedmulti(2,
  [fingerprint1/path1]xpub1.../0/*,
  [fingerprint2/path2]xpub2.../0/*,
  [fingerprint3/path3]xpub3.../0/*
))

# Taproot (single key)
tr([fingerprint/path]xpub.../0/*)

Key Functions

FunctionDescription
pk(KEY)Raw public key
multi(M, KEY, KEY, ...)M-of-N multisig (key order matters)
sortedmulti(M, KEY, ...)M-of-N multisig (keys sorted lexicographically)
sh(...)Wrap in P2SH
wsh(...)Wrap in P2WSH
tr(KEY) / tr(KEY, TREE)Taproot output (key-path, optional script tree)

Setting Up Different Accounts

Output descriptors use BIP 44/48/84/86-style derivation paths to separate accounts. The account index (third level) creates distinct address spaces from the same seed:

# Account 0 — e.g., main savings (2-of-3 P2WSH)
wsh(sortedmulti(2,
  [aabbccdd/48h/0h/0h/2h]xpub.../0/*,
  [eeff0011/48h/0h/0h/2h]xpub.../0/*,
  [22334455/48h/0h/0h/2h]xpub.../0/*
))

# Account 1 — e.g., spending wallet (2-of-3 P2WSH)
wsh(sortedmulti(2,
  [aabbccdd/48h/0h/1h/2h]xpub.../0/*,
  [eeff0011/48h/0h/1h/2h]xpub.../0/*,
  [22334455/48h/0h/1h/2h]xpub.../0/*
))
                          ▲
                    account index

Writing a New Descriptor

  1. Choose the script type: wsh(sortedmulti(...)) for P2WSH multisig, sh(multi(...)) for P2SH, tr(...) for Taproot.
  2. Gather xpubs: Export the extended public key from each hardware wallet at the correct derivation path:
    • P2SH multisig: m/45h or m/48h/0h/0h/1h
    • P2WSH multisig: m/48h/0h/0h/2h
    • Taproot: m/86h/0h/0h
  3. Include fingerprints: Each key should include the master fingerprint and derivation path in square brackets: [aabbccdd/48h/0h/0h/2h]xpub...
  4. Set the threshold: First argument to multi() or sortedmulti().
  5. Add the wildcard: Append /0/* for receive addresses, /1/* for change addresses. You need two descriptors per account (receive + change).
  6. Compute checksum: Use bitcoin-cli getdescriptorinfo "your_descriptor" to get the checksum-appended version.

9. Current Tools

Coordinator Software — Traditional Multisig (P2SH / P2WSH)

Sparrow Wallet

Full-featured desktop Bitcoin wallet. Best-in-class multisig coordinator — supports PSBT, output descriptors, P2SH, P2WSH. Connects to your own node via Electrum or Bitcoin Core. Taproot multisig not yet supported.

Blue Wallet

Mobile Bitcoin wallet (iOS/Android) with multisig vault support. Guided setup flow, PSBT round-trips via QR codes or files. Good for mobile-first multisig workflows.

Coordinator Software — Schnorr / Taproot Multisig

Nunchuk Beta

Taproot multisig wallet using MuSig2 (BIP 327) for key-path + script-path fallback. M-of-N up to 5 keys. Currently software (hot) keys only — hardware signer support planned. Launched Dec 2024.

Stack Wallet v2.0

Mobile wallet with FROST multisig via the modular-frost library (IETF-compliant, BIP-340 compatible). Built with Kayaba from the Serai project. Desktop coming soon.

Liana (Wizardsardine)

Taproot + Miniscript wallet with timelocked recovery paths. Working toward MuSig2 integration. Hardware support: Ledger (production), Coldcard (edge), BitBox02 (v8+), Specter DIY.

Bitkey (Block, Inc.) Shipping

Consumer 2-of-3 multisig: mobile key + hardware device + Block server. Uses MuSig2 for Taproot signing. Shipping since March 2024. Inheritance feature launched Jan 2025.

Hardware Wallets — Traditional Multisig

SeedSigner

DIY, air-gapped signing device (Raspberry Pi Zero). Stateless — no stored keys. QR-code PSBT signing. Open source. ~$50 in parts. Sovereignty-focused.

Keystone

Air-gapped hardware wallet with touchscreen. QR-code only communication. Supports multisig with Sparrow, Blue Wallet, and others. Open-source firmware.

Foundation Passport

Premium air-gapped hardware wallet. QR and microSD. Open-source firmware and hardware. Strong PSBT multisig support.

Coldcard (Mk4/Q)

Bitcoin-only hardware wallet. MicroSD/USB PSBT workflow. On-device multisig registration. Advanced: duress PINs, brick-me PIN, HSM mode. Edge firmware adds Taproot tapscript multisig (OP_CHECKSIGADD).

Hardware Wallets — Schnorr / Taproot Multisig

Frostsnap Shipping

Australian-built FROST hardware wallet. Devices daisy-chain via USB-C for distributed key generation — no single device ever holds the complete secret. Spends appear as standard Taproot single-sig on-chain. ESP32-C3 RISC-V, memory-safe Rust firmware. MIT licensed. 150k sats/device.

Team: Lloyd Fournier (cryptographer), Nick Farrow (engineer). Co-discoverers of the "Dark Skippy" attack on HW wallet nonce generation.

Ledger (Nano S+/X/Stax) Production

Most advanced MuSig2 support among traditional HW wallets. Bitcoin App v2.4.0 supports MuSig2 in key-path and script-path. Taproot-Miniscript in production. Works with Liana and Nunchuk.

Coldcard (Edge Firmware)

Edge firmware supports Taproot keyspend and tapscript multisig via sortedmulti_a (OP_CHECKSIGADD). BIP-388 miniscript format. Up to 32 signers. Not yet in stable firmware.

BitBox02

Taproot single-sig since 2022. Works with Liana for Taproot descriptors. ShiftCrypto actively exploring MuSig2/FROST but no shipping implementation yet.

Notable Implementations & Standards

libsecp256k1 v0.6.0

Upstream MuSig2 module (BIP 327) merged into the canonical library. Bitcoin Core PR #31244 implements MuSig2 descriptor parsing (BIP 390).

BitGo MuSig2 Production

First production MuSig2 taproot multisig wallet (2023). Three-path design: MuSig2 key-path (user+BitGo), tapscript fallbacks. Key-path: 57.5 vB vs 104.5 vB for native SegWit. Novel deterministic nonce scheme for stateless HSM signing.

ChillDKG (Blockstream)

Proposed BIP for standardized FROST distributed key generation. Based on SimplPedPop with pairwise ECDH. Active development, v0.2.0 Dec 2024.

BIP 445 — FROST Signing

Proposed BIP for FROST signing compatible with BIP 340 keys/signatures. Supports BIP 32 child key derivation and BIP 341 tweaking. Active development through 2026.
